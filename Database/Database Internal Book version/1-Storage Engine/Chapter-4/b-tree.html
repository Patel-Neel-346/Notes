Topics
Page Header
Every B-Tree page starts with a header containing metadata. The cell offset array keeps pointers in sorted key order, while cells themselves are stored in insertion order.
Key insight: Cells are written in insertion order (messy), but the offset array keeps their pointers in sorted key order. Binary search runs on the offset array — it never needs to rearrange the actual cells.
Sibling Links
Nodes at the same level can store direct pointers to their left and right neighbors. This avoids the costly round-trip up to the parent and back down when traversing siblings.
Navigation mode:
Without sibling links: must go up to parent then back down — potentially all the way to root if the sibling shares a higher-level ancestor.
Rightmost Pointer
In a B-Tree node, there is always one more pointer than keys. The last pointer has no paired key — it points to everything greater than the last key. Watch what happens when that rightmost child splits.
Why special? Every other pointer is paired with a separator key. The rightmost pointer covers the open-ended range (last_key, +∞) and must be stored separately — e.g. in the page header as SQLite does.
Overflow Pages
Pages are fixed-size (e.g. 4 KB). When a record is too large to fit, the first N bytes stay on the primary page and the rest spill into linked overflow pages.
Rule: max_payload_size = page_size / fanout. Any payload exceeding this spills to overflow pages. Keys rarely overflow (comparisons work on trimmed keys); data records may, but full retrieval is infrequent.
Binary Search with Indirection Pointers
Cells are stored in insertion order, but the offset array is sorted. Binary search runs on the offset array — each step picks the middle pointer, follows it to the cell, and compares the key.
Search for key:
Why not just sort the cells? Inserting a new cell in sorted position would require shifting all subsequent cells — expensive. The offset array is much smaller (just pointers), so re-sorting that is cheap.
Breadcrumbs (BTStack)
During a root-to-leaf traversal, each visited node is pushed onto a stack. If the target leaf needs to split, the stack is popped to find the parent and propagate changes upward.
PostgreSQL calls this stack BTStack. Each entry stores the node reference + the child pointer index followed. On split, pop the stack to locate the parent instantly — no tree re-traversal needed.
Rebalancing
Instead of immediately splitting an overfull node, elements can be redistributed to a sibling. This postpones expensive splits, improves node occupancy, and keeps the tree shallower.
B*-Trees (used by SQLite) push this further: keep distributing until both siblings are full, then split two nodes into three (each ⅔ full) rather than one into two (each ½ full).
Vacuum & Garbage Collection
Deletes only remove a cell's offset pointer — the cell data stays in place as unreachable garbage. Over time, pages become fragmented with no contiguous free space. Vacuum reclaims it.
After vacuum: dead cells are removed, live cells are rewritten in sorted key order, and the freed space is consolidated into a single contiguous block. IDs of fully freed pages are added to a freelist for reuse.
