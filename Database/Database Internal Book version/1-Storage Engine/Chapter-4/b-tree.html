DATABASE STORAGE ENGINE

Topic 01
Page Header
Every B-Tree page begins with a fixed-size header. The cell offset array holds pointers in sorted key order, while actual cell data is written in raw insertion order — messy but fast.

This separation is the core trick: binary search runs on the tiny sorted array (cheap), not the large cell data. Hover each pointer in the offset array to trace where it points.
Why this design? Cells are expensive to move (can be hundreds of bytes). The offset array holds only 2-byte pointers, so re-sorting it on insert costs almost nothing. Binary search operates on the array — O(log n) comparisons on a tiny array, not on raw cell data. This is how SQLite and PostgreSQL both work.
Topic 02
Sibling Links
Leaf nodes at the same level keep a direct pointer to their right neighbor. Without it, scanning from one leaf to the next requires climbing all the way up to the parent and descending again — wasting precious I/O reads. This is why range scans are fast in B+ Trees.
Range scan example: SELECT * WHERE key BETWEEN 10 AND 100. The engine finds key=10 via tree traversal, then follows sibling links rightward across all leaf pages. Without these links, each step would require O(log n) page reads. With them, it's O(1) per leaf hop. This is the defining advantage of B+ Trees over B-Trees.
Topic 03
Rightmost Pointer
Think of keys as fence posts and pointers as fields between them. With 3 fence posts you get 4 fields — but the last field has no right boundary.

The rightmost pointer is that last boundless field: it catches everything greater than the last key, with no upper limit. Watch how it shifts when the node splits.
SQLite explicitly labels this field "rightmost pointer" in its page header struct. When a split occurs, the highest key of the left half becomes a new fence post — and the rightmost pointer simply shifts to point to the new rightmost child. The open range shrinks but never disappears.
Topic 04
Overflow Pages
Pages have a fixed size (typically 4 KB or 8 KB). When a single cell's payload is too large, it spills into a chain of linked overflow pages. The original cell keeps only the first N bytes plus a page ID pointing to the chain head.
Performance impact: Each overflow page hop is an additional disk read. Reading a cell spread across 3 overflow pages costs 4 I/Os total. This is why databases encourage storing large BLOBs externally (filesystem) and keeping column values small. Keys rarely overflow because they use trimmed comparison keys.
Topic 05
Binary Search with Indirection
The offset array is sorted by key, but cell data is scattered. Binary search works on the array of pointers — picking the middle pointer, following it to the cell, reading its key, and narrowing down. Pick a target key, then step through the algorithm.

Complexity: O(log n) comparisons on the sorted offset array. Each comparison follows one pointer to read a key. The cell data itself is never rearranged. Inserting a new cell appends it to the end of cell space and just inserts one entry into the sorted offset array — O(n) shift of tiny 2-byte values, not O(n) shift of full records.
Topic 06
Breadcrumbs (BTStack)
When descending the tree to insert a key, the engine pushes every visited node onto a stack — like dropping breadcrumbs. If the target leaf needs to split, the engine pops the stack to instantly find the parent without re-traversing the tree. PostgreSQL calls this BTStack.
Why the stack matters: Splits can cascade. Leaf splits → parent needs a new key → parent might split too → its parent might split → all the way to root. The BTStack lets each split operation instantly pop to find its parent without rescanning. In the worst case, stack depth = tree height = O(log n).
Topic 07
Rebalancing
Before splitting an overfull node, the engine checks if a sibling has room. If it does, keys are redistributed through the parent — no new page needed. This improves fill factor and keeps the tree shallower. SQLite's B*-Tree variant tries even harder: it defers splits until two siblings are both full.
Key rotation: The largest key from the left (38) moves up to replace the separator (40) in the parent. The old separator (40) moves down into the right node. This maintains the sort invariant throughout and avoids allocating a new page.
Topic 08
Vacuum & Garbage Collection
Deleting a record doesn't erase cell data — it only removes the cell's entry from the offset array. The data sits as invisible garbage. Over time, pages become fragmented. VACUUM rewrites live cells contiguously, reclaiming the dead space.
VACUUM in practice: SQLite's VACUUM rewrites the entire database. PostgreSQL's AUTOVACUUM runs in the background to reclaim dead tuples. Fully empty pages are returned to the freelist. Without VACUUM, repeated INSERT/DELETE cycles lead to massive disk bloat — the database file grows but utilization drops.
